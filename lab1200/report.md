#### 2. Реактивная загрузка списка (ViewModel)

```kotlin
private fun loadNotes() {
    viewModelScope.launch {
        _isLoading.value = true
        try {
            repository.allNotes
                .onEach { notesList ->
                    _notes.value = notesList
                    if (_isLoading.value) _isLoading.value = false
                }
                .launchIn(viewModelScope)
        } catch (e: Exception) {
            _error.value = "Ошибка загрузки заметок"
            _isLoading.value = false
        }
    }
}
```

### Пример работы программы

1. **Главный экран** — отображается список заметок с заголовком, содержанием (первые 2 строки) и датой создания в формате `ДД.ММ.ГГГГ, ЧЧ:ММ`. Кнопка «+» в правом нижнем углу.

1. Применение Flow и StateFlow гарантирует автоматическую синхронизацию экрана при любых изменениях данных в базе (добавление, удаление, редактирование). Это исключает необходимость в ручном обновлении списка.

## Ответы на контрольные вопросы

1. **В чем преимущество использования Flow и StateFlow перед обычными списками?**  
   `Flow` и `StateFlow` работают в реактивной парадигме: они самостоятельно оповещают подписчиков (например, UI) о любых изменениях данных, что убирает необходимость вручную вызывать обновление списка. `StateFlow` дополнительно хранит текущее состояние и легко интегрируется с Compose.

2. **Почему ViewModel не уничтожается при повороте экрана и как это влияет на UX?**  
   `ViewModel` привязана к жизненному циклу `Activity` и переживает повороты. Это позволяет сохранять загруженные данные, введённый текст и общее состояние интерфейса, пользователь не теряет прогресс при смене ориентации устройства.

3. **Какие преимущества дает использование Room по сравнению с прямым использованием SQLite?**  
   `Room` проверяет SQL-запросы на этапе компиляции, что снижает число ошибок в рантайме. Он автоматически маппит объекты в таблицы и обратно, поддерживает корутины и `Flow`, упрощает миграции и сокращает объём шаблонного кода.
